# 赛题需求与证据矩阵

## 目的与边界

本矩阵审计 LiteTest-MAS 当前代码、测试、M9 正式公开结果和 M10 离线 Dashboard。M9 的唯一有效冻结为 `cc7aac0417afb6acab47baaf7449459692fa9444`，其 240 条正式记录、公开聚合、Dashboard 和发布标签 `v1.0.0-experiment` 均为只读基线。

状态只使用以下五种取值：`已充分证明`、`已实现但证据不足`、`部分实现`、`未实现`、`不适用`。`unavailable` 表示既有公开记录无法恢复某项指标，不代表零值，也不能据此推测结果。

现有 Dashboard 页面指 M10 的离线“结果总览”“组别结果”“配对比较”视图；“无”表示当前没有对应页面。

## M9 可恢复性清单

既有 M9 公开记录可直接恢复：`message_count`、`text_character_count`、`protocol_event_count`、`prompt_tokens`、`completion_tokens`、`total_tokens`、`request_count`、`latency_seconds`、`state_vector_count`、`state_vector_bytes`、`memory_reference_ids`、`memory_read_count`、`memory_hit_count`、`memory_reuse_count`、`memory_write_count`、质量和 Sandbox 字段。

以下指标在既有 M9 公开记录中为 `unavailable`：`agent_text_message_count`、`agent_protocol_message_count`、`agent_text_tokens`、`protocol_payload_bytes`、`protocol_payload_tokens`、`protocol_header_bytes`、`capability_handshake_count`、`capability_handshake_bytes`、`reference_id_count`、`repeated_context_bytes`、`deduplicated_context_bytes`、`state_reference_bytes`、`equivalent_text_state_bytes`、`state_compression_ratio`、`state_encode_latency`、`state_decode_latency`、`memory_candidate_count`、`memory_accept_count`、`memory_reject_count`、`memory_injected_count`、`memory_injected_tokens`、`memory_injected_bytes`、`memory_success_write_count`、`memory_eviction_count`、`memory_accept_rate`、`memory_effective_reuse_rate` 和 `total_wall_time`。

P2 只能对可恢复字段重新聚合；对上述字段必须输出 `unavailable` 和缺失原因，不得填 `0`，不得从总模型 Token、文本字符或质量结果反推。

## 当前结论

M9 已充分证明固定任务、固定模型、固定配置下 Protocol V1 的质量变化及总模型成本变化，也证明了 StateVector V1 和 SharedMemory V1 在 Runner 中被执行并留下基础计数。它尚未证明 Protocol V1 节省 Agent 间通信、StateVector V1 替代重复文本，或 SharedMemory V1 的复用带来因果质量收益。G4 相对 G3 的任务成功率下降必须作为潜在负迁移保留。
